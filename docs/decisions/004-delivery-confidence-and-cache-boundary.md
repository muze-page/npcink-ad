# ADR 004: Define delivery confidence and cache boundaries

- Status: Accepted
- Date: 2026-07-15

## Context

ADR 003 established one `npcink_promotion` record as the complete publishing unit. The next delivery-confidence work adds list status, publication preflight, real-page preview, and scheduled delivery. These surfaces must explain the same rules without overstating what the plugin can know about a particular visitor request or third-party page cache.

A manual promotion block may be inserted in post content, a synced pattern, or a template, and those sources may change independently. Scanning every page cannot prove that the block will exist in every runtime context. Likewise, PHP cannot enforce a schedule boundary while a full-page cache or CDN continues serving HTML generated by an earlier request.

## Decision

The single Promotion model remains the Npcink Ad 0.1 baseline. `Eligibility_Evaluator` is the only source of eligibility rules for frontend delivery, the promotion list, and publication preflight:

- frontend delivery evaluates the promotion against the real request context;
- the list reports configuration and temporal rule status, but must say **rules ready**, never that a promotion is currently displaying;
- publication preflight uses the same evaluator and stable reason codes instead of duplicating validation in UI or REST code.

Presentation layers may translate or group reason codes, but must not create independent eligibility rules.

“Non-empty creative” means content with a meaningful rendered payload, not merely a non-empty serialized string. Whitespace, HTML comments, empty text blocks, and empty layout containers are invalid; visible text, media elements, background imagery, and dynamic blocks that may render at runtime remain valid. The editor mirrors this rule, while the PHP evaluator is authoritative.

The supported management write paths are the block editor's protected Core REST controller and the list's dedicated pause/resume action. Native Quick Edit and Bulk Edit are not offered for Promotions because they cannot submit the complete creative and typed delivery rules for the same preflight decision.

For `block` delivery, the plugin does not scan the whole site or claim that an entrypoint exists everywhere. Preflight warns that the block must be verified in context, but a missing or unverified block does not prevent publication. Previewing the selected real page is the final evidence for whether the promotion is inserted and eligible in that context.

Without full-page caching, the evaluator applies schedule boundaries on every request: start is inclusive and end is exclusive. A promotion is temporally eligible when `start <= now < end`, with either absent boundary treated as open.

With a full-page cache or CDN, minute-level switching cannot be guaranteed by the plugin alone. Operators must use a cache TTL that reaches the next boundary or purge the affected pages at that boundary. Npcink Ad 0.1 does not include a cache-provider adapter, purge queue, or cron scheduler.

## Acceptance matrix

| Surface or condition | Expected result |
| --- | --- |
| Invalid content, targets, or schedule | Frontend, list, and preflight derive the same reason codes from `Eligibility_Evaluator`. |
| Serialized content only contains comments or empty layout/text blocks | Publication is rejected as empty creative. |
| A non-empty schedule value is not a real calendar datetime | Publication is rejected as an invalid schedule; it is not converted to an open boundary. |
| Published and structurally ready promotion | List says rules are ready; it does not claim a visitor is seeing the promotion. |
| Manual block is not globally verifiable | Preflight shows a contextual warning and still permits publication. |
| Selected real page has no manual block | Real-page preview is the evidence of the missing entrypoint; no site-wide scan is attempted. |
| Request occurs exactly at `start` without full-page caching | Promotion may render when all other rules pass. |
| Request occurs exactly at `end` without full-page caching | Promotion is expired and does not render. |
| Full-page cache or CDN serves older HTML across a boundary | No minute-accuracy claim is made; TTL or boundary purge is required. |
| Authorized preview | Uses the same rules, reports the truthful verdict, and is non-cacheable. |

## Alternatives considered

### Infer live delivery from the promotion list

Rejected. The list lacks a specific page, block insertion point, device simulation, and cache state, so “currently displaying” would be unverifiable.

### Scan all content for manual blocks before publication

Rejected. A scan is expensive and still cannot prove runtime insertion through templates, patterns, filters, or later edits. Real-page preview provides stronger contextual evidence.

### Add cron and cache integrations now

Rejected. Cache providers have different purge contracts, and cron execution is not exact. Adding adapters before a deployment requirement exists would create operational complexity without a reliable universal guarantee.

## Consequences and extension boundaries

- New delivery surfaces must call `Eligibility_Evaluator`; they may not fork its rules.
- A future cache integration must define provider capability detection, affected-URL resolution, idempotent purge behavior, failure observability, and retry semantics in a new ADR.
- Cron may coordinate a known cache integration, but must not become a second publication or eligibility state.
- Schedule guarantees must be stated per deployment and verified against the active cache path.
- A separate placement, rotation, or reusable-slot model still requires observed demand and a new decision; it is not implied by cache work.
- Site-wide block indexing may be added only as advisory evidence with explicit freshness limits, never as a publication gate or replacement for real-page preview.
