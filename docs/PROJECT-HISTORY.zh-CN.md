# Npcink Ad 项目历史、开发复盘与后续原则

- 状态：当前复盘文档
- 更新日期：2026-07-18
- 覆盖范围：从 Magick AD 原型到 Npcink Ad `0.3.2` 首发候选
- 当前发布状态：GitHub Release `v0.3.2` 已发布；尚未提交 WordPress.org 审核

## 1. 文档用途与权威边界

这份文档解释项目为什么从一个功能广泛的广告管理原型，重构成一个聚焦的
WordPress 原生站内推广发布工具。它用于帮助未来维护者和代理理解历史、避免
重复争论和重新引入已经否决的复杂度，不代替运行代码或正式决策记录。

项目事实按以下优先级判断：

1. 运行代码和自动化测试是当前行为真相；
2. [产品契约](product-contract.md)、[架构总览](architecture-overview.md)和
   [ADR](decisions/)定义当前产品与架构边界；
3. 发布候选记录、Git 标签和发布产物定义某个版本的发布状态；
4. 本文与 `docs/archive/` 解释历史，不得用来恢复已删除的运行契约；
5. 聊天结论、截图和本地演示数据只有在被代码、测试、ADR 或发布回执确认后，
   才能升级为项目事实。

特别需要区分三个 WordPress.org 目录状态：

- **已准备提交**：资料和 ZIP 已就绪，但尚未上传；
- **审核中**：已经获得 WordPress.org 提交回执并进入审核队列；
- **已发布**：审核通过、SVN 与公开下载均已验证。

截至本文更新时，Npcink Ad 的 WordPress.org 首发只达到第一种状态。

## 2. 一句话结论

项目真正的进步不是把 Magick AD 的功能全部搬回来，而是把旧项目积累的产品
知识，压缩成一条可以被普通 WordPress 编辑者理解、验证和完成的工作流：

> 创建 Promotion → 设置投放规则 → 在真实页面查看诚实结论 → 发布或暂停。

重构后的核心价值来自低门槛、WordPress 原生、无访客追踪、统一判定和真实页面
验证，而不是功能数量。

## 3. 起点：为什么 Magick AD 需要重置

2026 年 1 月至 2 月的 Magick AD 原型快速探索了后台 SPA、广告组与位置、模板、
复杂定向、统计追踪、队列、Cron、A/B、CMP、调试界面和多个前端运行时。它证明
了很多能力在技术上可做，但没有先证明用户能否轻松完成第一条有效投放。

归档标签 `archive/magick-ad-pre-npcink` 中，按根入口与 `src` PHP、`assets/js`
同一口径统计，运行 PHP 约 16,086 行、前端源码约 18,678 行；当前实现分别约为
5,897 行和 5,511 行，下降约 63% 和 70%。代码行数本身不是质量指标，但这组变化
证明被删除的是整套未验证子系统，而不只是改名或移动文件。旧实现形成了几个问题：

- 用户必须理解广告、广告组、位置、模板、规则、变体和统计等多组概念；
- 后台状态、前端判定和统计运行时各自拥有部分真相；
- 自定义表、公开追踪入口、队列和缓存把隐私、可靠性与卸载成本提前引入；
- 巨型控制器、巨型设置组件和多套投放路径提高了修改与验证成本；
- 大量界面优化发生在主工作流尚未被真实用户验证之前。

项目没有投入使用、没有真实数据和公共兼容承诺，因此主要矛盾不是“如何兼容旧
系统”，而是“如何用最小复杂度证明一条可靠的发布工作流”。继续增量修补会把
未验证的原型永久化，破坏性重置反而是成本最低的选择。

## 4. 演进阶段

### 4.1 保存研究成果，破坏性重置运行时

2026-07-15，旧实现被保存在 `archive/magick-ad-pre-npcink`，随后通过
`bf617be` 和 [ADR 001](decisions/001-pre-ga-native-wordpress-baseline.md) 重置为
原生 WordPress 基线。

这一阶段删除了统计表、追踪、队列、Cron、CMP、A/B、模板迁移、自定义管理 SPA、
旧 REST 控制器和兼容层，保留的不是旧代码结构，而是几个经过验证的方向：服务端
渲染、可测试的投放资格、受权限保护的管理操作和可验证的发布包。

### 4.2 重命名并收缩为单 Promotion 工作流

[ADR 002](decisions/002-npcink-ad-product-direction.md) 确立 **Npcink Ad** 品牌、
`npcink-ad` slug/text domain 和 WordPress 原生站内推广定位；
[ADR 003](decisions/003-single-promotion-record.md) 进一步删除 Promotion 与 Placement
双记录模型，只保留一个 `npcink_promotion` 记录。

标题、区块创意、发布状态、位置、内容范围、排期和设备规则归属于同一记录，
`post_status` 成为发布/暂停的唯一状态真相。`ade9e04` 建立单 Promotion 流程，
`bc04806` 补齐简体中文，`cb05233` 把这组工作收口为新的 beta 基线。

### 4.3 先解决“为什么不展示”，再扩展位置

真实使用中最重要的问题不是再增加一种广告，而是避免后台说“已发布”、前台却
没有展示。`12d2683`、`2e5768d` 和
[ADR 004](decisions/004-delivery-confidence-and-cache-boundary.md) 建立了统一的
`Eligibility_Evaluator`、稳定 reason codes、发布前预检、列表状态、真实页面预览、
快捷暂停/恢复和明确的缓存边界。

从此以后，列表、编辑器、预览和正式前台不得各写一套规则。管理界面只能翻译
判定器结论，不能发明新的“已投放”状态。时间边界在动态请求中可精确执行，但全页
缓存或 CDN 可能继续提供旧 HTML；插件必须提示 TTL/purge 边界，不能承诺穿透任意
缓存实现分钟级切换。

### 4.4 有边界地吸收旧项目中真正有用的能力

在不恢复旧系统的前提下，项目逐步加入直接服务主工作流的能力：

- `5351a8f` / [ADR 006](decisions/006-paragraph-anchor-delivery.md)：第 1–20 段后投放；
- `4efafa6` / [ADR 007](decisions/007-canonical-editorial-scope.md)：互斥的标准内容范围；
- `18449f1` / [ADR 008](decisions/008-manual-placement-and-device-guidance.md)：手动区块、
  专家短代码和固定设备断点；
- `033c1a8`：可靠的 Promotion 搜索、分页和已保存 ID 解析；
- `3f11eb0` / [ADR 009](decisions/009-split-editor-entrypoints.md)：拆分普通区块编辑器与
  Promotion 编辑器资产，避免无关页面加载；
- `94726c2` / [ADR 010](decisions/010-site-controlled-video-creative.md)：把 Core Video 当作
  同一 Promotion 的素材，而不是建立第二套视频广告系统；
- `05aafe1`：改进排期编辑、分类和标签选择；
- `a404fd1` / [ADR 011](decisions/011-bounded-page-bar-delivery.md)：正常文档流中的顶部/
  底部横栏，而不是恢复悬浮容器运行时；
- `9a340e0`：移动端触控目标、RTL 间距和横栏模板收口。

这些能力都复用同一 Promotion、同一判定器、同一预览与同一发布状态。项目吸收的
是旧原型证明有价值的用户能力，不是旧原型的耦合结构。

### 4.5 从“能用”转向“容易第一次成功”

右侧编辑器侧栏曾经同时承载大量设置和突兀的错误提示。后续通过 `64549c5`、
`96d0eac` 和 `0bcf772` 完成三项修正：

- 空列表和未完成 Promotion 提供由现有状态派生的三步引导，不保存新的向导状态；
- 错误、警告和普通说明建立信息层级，避免所有提示都以高强度警报出现；
- 大量投放设置移入分 Tab 的模态框，侧栏只保留摘要和主要动作。

五人真实试用没有发现重复的首投阻断问题后，项目才进入横栏能力阶段。这说明
“用户完成任务”比“界面看起来功能丰富”更适合作为扩展门槛。

### 4.6 发布工程和 WordPress.org 首发准备

`v0.2.0`、`v0.2.1`、`v0.2.2` 和 `v0.3.1` 逐步验证了功能与发布门禁；
`82df4e0` / `v0.3.2` 完成 WordPress.org 文案、图标、横幅、截图、标准语言包边界、
官方 Plugin Check 和唯一发布产物。

`049d1b5` 又加强了 Playground 启动重试和 fixture 锁，说明发布工程本身也需要像
运行代码一样处理竞态与可重复性。`v0.3.2` 的唯一 ZIP SHA-256 是：

```text
0ccaacf5452c316184e852180ce7bf9fb7785f36b9db26f59eed0725512e94de
```

同一版本的 GitHub Release、WordPress.org 首次上传和未来 SVN tag 必须使用这个
已校验产物，不得在本地重新压缩后继续冒用 `0.3.2`。

### 4.7 完成品牌的本地开发迁移

代码、namespace、text domain、仓库和发布包已经完成 Npcink Ad 改名后，本地源码
目录最后从旧项目名改为 `npcink-ad`，`bc88efa` 同步清理编辑器任务标签和本地杂项。

这次迁移暴露了两个通用问题：

- WordPress 测试站若通过软链接挂载插件，源码目录改名后必须同步更新链接；
- 移动 pnpm 项目后，即使锁文件未变，已有 `node_modules` 布局也可能需要通过
  `pnpm install --frozen-lockfile` 重建。

软链接站点适合日常开发，但不能代替发布 ZIP 的干净安装验证。

## 5. 从旧项目吸收了什么，没有吸收什么

### 选择性吸收

- 真实主题页面中的桌面/移动预览；
- 用户能理解的运行结论和不展示原因；
- 正文前、正文后、段落后、手动区块以及有边界的顶部/底部横栏；
- 标准文章/页面、显式内容以及 Core 分类/标签范围；
- Core 区块、媒体库和 Video 素材；
- 对主题挂钩、缓存、排期和设备边界的经验；
- 可重复的发布包、Playground、E2E 和 Plugin Check 门禁。

### 明确不恢复

- 旧后台 SPA、广告组、Slot/Resolver 和模板市场；
- 统计追踪、访客画像、Cookie、事件队列和自定义统计表；
- A/B、轮播、权重、频控、地理定向和 CMP；
- Popup、插屏、悬浮/吸顶容器和持久关闭状态；
- VAST、贴片视频、播放分析和广告网络协议；
- 任意 PHP/JavaScript 注入、兼容报告和调试一级页面；
- 未发布数据、旧接口和旧标识的兼容层。

判断标准不是“旧项目是否已经写过”，而是“它是否明显降低正确发布一条 Promotion
的时间或不确定性，并且由 Npcink Ad 自己维护比复用 WordPress/成熟插件更合理”。

## 6. 已验证的开发思路

### 6.1 先定义用户任务，再决定数据模型

一个用户任务对应一个 Promotion，而不是先设计多个可复用实体再让用户理解它们。
只有真实需求证明一条创意必须拥有多个独立发布状态时，才重新讨论 Placement。

### 6.2 一个判定器服务所有状态表面

列表、预检、真实页面预览和正式投放必须共享 PHP 判定器和 reason codes。浏览器
可以即时镜像规则以改善反馈，但 PHP 始终是发布与投放的最终依据。

### 6.3 优先使用 WordPress 已有能力

内容使用区块编辑器，素材使用媒体库和 Core Video，状态使用 `post_status`，数据
使用 CPT 与 typed meta，权限使用 capability，管理读写优先使用 Core REST。
只有现有能力无法表达已验证需求时才增加自有系统。

### 6.4 把产品承诺写成可验证边界

“排期可用”必须同时说明缓存限制；“移动端适配”必须给出正式断点；“视频可用”
必须区分结构有效与网络可播放；“预览”必须区分强制显示素材与真实 eligibility。
诚实的小承诺比无法验证的大承诺更可靠。

### 6.5 用渐进披露控制复杂度

默认界面只展示主流程、摘要和下一动作。密集设置进入模态框并按内容、位置、范围、
排期等 Tab 组织；高级说明不能抢占主要操作，错误强度必须匹配真实严重程度。

### 6.6 新功能默认拒绝，按证据准入

每项提案必须回答：

1. 它是否减少正确发布 Promotion 的时间或不确定性？
2. WordPress 或成熟插件已经提供时，为什么仍必须由 Npcink Ad 拥有？
3. 能否继续使用单 Promotion、统一判定器和现有产品语言？
4. 权限、隐私、缓存、卸载、翻译和测试契约是否明确？
5. 是否已有重复用户需求，而不是只因技术上可做？

任何一项没有答案，就不进入核心产品。

### 6.7 中文不是发布末期补丁

界面设计、错误层级、日期时间和发布文案必须同时验证中文。新增字符串后立即更新
POT/PO/MO/JSON，并让翻译完整性成为门禁，避免功能完成后再被迫重做布局。

### 6.8 发布物是不可变证据

版本号、标签、源码提交、ZIP 和校验和是一组原子事实。发布后不得重建同名 ZIP；
需要修改运行代码时创建新版本并完整复跑门禁。WordPress.org 的“已提交/审核中”
必须由外部回执证明，不能从本地准备状态推断。

### 6.9 本地源码验证与发布包验证必须分开

软链接可以证明当前源码被 Local 加载，却不能证明 WordPress 能安装发布 ZIP。
发布前必须在干净站点安装准确 ZIP，并对同一文件运行 Plugin Check、包内容检查和
烟雾测试。

### 6.10 多代理只用于可分割工作

并行代理应先划分文件区域和验收接口：例如 PHP 领域层、编辑器 UI、E2E/文档分别
负责，避免同时修改同一文件。主代理负责产品边界、跨层契约、最终合并和 staged
diff 验证；并行不能替代统一集成测试。

## 7. 工作审视报告

### 原定目标

在没有兼容负担的前提下，系统审计旧插件，降低上手门槛，明确相对成熟插件的价值，
完成 Npcink Ad 品牌与中文支持，选择性吸收必要的投放能力，并准备可审核的
WordPress.org 首发版本。

### 完成情况

- [x] 将旧原型保存为研究档案，破坏性删除未验证的复杂系统；
- [x] 建立单 Promotion、WordPress 原生、隐私优先的完整主流程；
- [x] 统一列表、预检、预览和前台投放判定；
- [x] 通过真实用户试用修正提示层级、首次引导和侧栏密度；
- [x] 有边界地加入段落、分类/标签、设备、视频和页面横栏；
- [x] 补齐简体中文、测试矩阵、Plugin Check、发布包和 WordPress.org 资料；
- [x] 完成 GitHub 仓库、发布包和本地开发目录的品牌迁移；
- [x] 完成准确 `v0.3.2` ZIP 的校验、Plugin Check、干净安装和浏览器烟雾验收；
- [ ] 提交 WordPress.org 并取得审核回执；
- [ ] 审核通过后完成 SVN、公开下载与目录页面验证。

后两项需要外部提交或审核证据，不能提前宣称完成。

### 发现的问题

| 严重程度 | 具体问题 | 根本原因 | 改进方法 |
| --- | --- | --- | --- |
| 必须改正 | 曾把“WordPress.org 资料已准备”表述成“正在等待审核”，但实际上尚未提交 | 发布状态没有绑定外部回执，聊天阶段结论被当成事实 | 使用明确发布状态机；没有上传回执就只能写“准备提交” |
| 应当改正 | 早期 Magick AD 在首条投放未验证前就加入 SPA、追踪、统计、A/B、CMP 和多套运行时 | 以功能覆盖率代替用户任务完成率 | 先写产品契约和三分钟成功预算，再决定是否新增实体或系统 |
| 应当改正 | 多次“下一阶段”讨论在功能扩展与发布收口之间切换，存在重新膨胀风险 | 没有持续执行单一在制品限制 | 一个阶段只允许一个主目标；进入 release freeze 后只处理阻断问题 |
| 应当改正 | 早期右侧栏承载太多设置，错误提示强度过高 | 数据字段直接映射为界面，没有信息密度和反馈层级预算 | 先设计主动作和摘要，再用模态框/Tab 渐进披露；警报只用于阻断问题 |
| 应当改正 | 截图和 E2E 曾修改本地演示数据，早期记录没有完整证明全部恢复 | fixture 创建、截图和清理没有被视为一个原子测试 | 使用幂等 fixture、唯一锁、`trap` 清理和前后状态快照 |
| 建议改进 | 品牌代码改名完成后，本地源码目录较晚才改名，导致软链接与 pnpm 依赖布局需要额外处理 | 改名检查清单只覆盖仓库内容，没有覆盖开发环境 | 品牌迁移同时检查远程、目录、IDE、软链接、依赖目录、ZIP 和 CI |
| 建议改进 | 通用 WordPress triage 对仓库根目录插件返回 `unknown` | 分类器偏向 `wp-content/plugins/` 形态 | 始终交叉检查插件头、`composer.json`、`block.json`、入口文件和脚本 |

### 做得好的地方

- 在没有用户和兼容负担时真正删除旧系统，而不是保留双写与迁移税；
- 用户反馈能迅速转化为具体修正，例如状态真相、提示层级、模态框和时间格式；
- 每次能力扩展都逐步补齐 ADR、稳定 reason code、翻译和跨层测试；
- 明确拒绝追踪和访客状态，形成了清晰的隐私与缓存优势；
- 使用真实主题、真实页面、最低/当前 WordPress、移动端和五人试用补充单元测试；
- Git 标签、GitHub Release、ZIP 和校验和形成了可追溯的发布证据。

### 下次重点关注

- 任何外部发布状态都必须记录回执、时间和准确产物；
- release freeze 期间不加入新功能；
- 在软链接开发环境之外，始终验证准确发布 ZIP；
- 新功能先收集重复需求，再写 ADR 和验收矩阵；
- 多代理集成前先检查文件所有权，集成后检查 staged diff 和完整门禁。

## 8. 今后的标准工作流

### 产品提案

1. 记录真实用户问题和出现频次；
2. 判断 WordPress Core 或成熟插件是否已经解决；
3. 用产品契约准入问题过滤；
4. 如需改变长期边界，先写 ADR；
5. 定义最小纵向切片和明确非目标。

### 实现

1. 先扩展领域契约和表驱动测试；
2. 再接入 Repository、Preflight、List、Preview 和 Delivery；
3. 管理界面只展示统一领域结果；
4. 同步更新中文翻译、架构文档和发布包契约；
5. 多代理按文件区域并行，由一个负责人完成集成。

### 验证

1. `composer check`；
2. `pnpm check`；
3. 最低与当前 WordPress/PHP Playground；
4. 经典主题与区块主题矩阵；
5. 打包插件 Gutenberg E2E；
6. Local 真实主题/页面烟雾测试；
7. 准确 ZIP 的 Plugin Check 和干净安装；
8. `git diff --check`、staged diff 和发布产物校验和。

### 发布状态机

```text
设计通过
  → 代码完成
  → 本地与自动化验证通过
  → 标签及唯一产物发布
  → 准确产物独立验收
  → WordPress.org 已提交（有回执）
  → 审核中
  → 已批准
  → SVN 与公开下载已验证
```

每一步只能由对应证据推进，不能凭计划或口头同意跳级。
