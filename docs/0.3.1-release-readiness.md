# Npcink Ad 0.3.1 发布候选就绪记录

- 状态：Release Candidate
- 日期：2026-07-17
- 发布渠道：GitHub prerelease

## 目标与非目标

0.3.1 的目标不是继续增加广告类型，而是把已经通过真实使用的单一
Promotion 工作流变成可重复、可审计、可回滚的首个公开候选版本。本阶段只收口
许可证、源码披露、官方检查、主题兼容、缓存边界和发布产物。

本阶段不增加 Popup、悬浮/吸顶横栏、VAST、统计、频控、地理定向、第二套数据模型
或新的一级管理页面。

## 发布契约

- 插件头、PHP 常量、`package.json`、`readme.txt` Stable tag 与 Git 标签必须使用同一版本。
- 插件声明 GPLv2 or later，仓库与发布包携带完整 GPLv2 文本。
- WordPress.org Contributors 使用已存在的账号 `muze233`；Npcink 继续作为产品品牌和作者名。
- `readme.txt` 公开完整源码仓库、Node.js 20 / pnpm 10 构建方法，以及 Composer 依赖只用于开发检查的事实。
- 每个发布标签只产生一个 `npcink-ad-<Version>.zip` 和对应的 `.zip.sha256`。
- 标签提交必须包含在 `origin/master` 历史中；标签触发的 GitHub Actions 必须先完成全部门禁，再创建或更新 GitHub prerelease。
- 同一标签的工作流串行执行；若该标签以后已转为正式 Release，重跑只能保持原状，不能降回 prerelease。

| 官方规则 | WordPress Plugin Check 2.0.0 不得报告 error | `tests/plugin-check/run.sh` |
| WordPress | 最低 WP 6.5/PHP 8.1 与当前 WP/PHP 组合 | `tests/playground/run.sh` |
| 编辑器 | Add New 到发布、真实页预览、暂停和恢复 | `tests/e2e/run.sh` |

| 缓存 | PHP 请求遵守开始、停止、暂停、恢复；整页缓存继续遵守公开边界 | [ADR 004](decisions/004-delivery-confidence-and-cache-boundary.md) |

主题矩阵固定使用以下官方发布包并校验下载 SHA-256，避免“latest”在同一提交上产生
不同结果：

- Twenty Twenty-One 2.8，WordPress 6.5 / PHP 8.1；
- Twenty Twenty-Five 1.5，WordPress 7.0 / PHP 8.5。

两种主题都必须证明顶部和底部横栏只渲染一次、保持正常文档流、关闭按钮至少
44 × 44 CSS 像素、390px 视口没有横向溢出，而且关闭其中一条不会影响另一条。

## Plugin Check 警告分类

Plugin Check 2.0.0 在 WordPress 6.5.8 的完整 `new` 模式中报告零 error 和三条
warning。warning 保持可见，不在 CI 中隐藏；新增 error 会阻断发布。

| 警告 | 决定 | 理由与后续边界 |
| --- | --- | --- |
| `PluginCheck.CodeAnalysis.DiscouragedFunctions.load_plugin_textdomainFound` | 接受 | GitHub/本地 ZIP 安装仍需从插件内 `languages/` 加载完整简体中文包；调用位于 `init`，不提前触发翻译。若改为只通过 WordPress.org 语言包分发，再移除。 |
| `WordPress.DB.SlowDBQuery.slow_db_query_meta_query` | 接受并监测 | 当前查询只读取已发布的单一 Promotion 类型，关闭分页计数并一次预热 meta；在没有真实规模证据前，不为消除静态警告引入自定义表或第二套索引。若真实站点出现查询瓶颈，再用测量结果决定索引。 |
| `WordPress.NamingConventions.PrefixAllGlobals.NonPrefixedConstantFound` | 接受 | 被报告的 `DONOTCACHEPAGE` 是缓存插件共同识别的既有全局常量；改名会失去禁止缓存的语义。代码只在授权预览请求中、且仅在尚未定义时设置。 |

## 缓存与真实试运行边界

[ADR 004](decisions/004-delivery-confidence-and-cache-boundary.md) 已在 `ad.local` 的
WP Super Cache 3.1.1、60 秒 TTL 下验证开始、停止、暂停和恢复：PHP 动态请求立即使用
统一判定器，缓存 HTML 在 TTL 或 purge 前保持旧结果。0.3.1 的横栏同样由服务端
判定器输出，没有新增客户端投放状态，因此不扩大缓存承诺。

候选版本在真实站点试运行时必须记录：站点主题、缓存实现与 TTL、至少一条正文位置、
一条页面横栏、一次暂停/恢复，以及一次跨开始或停止边界的结果。若无法为具体缓存环境
定义 TTL 或受影响页面 purge，必须保留后台明确提示，不能宣称分钟级准时切换。

2026-07-17 又在 `ad.local` 复验了当前候选源码：WordPress 7.0.1、PHP 8.2.29、
Twenty Twenty-Five 1.5，且当时没有 advanced-cache drop-in。临时创建的所选页面同时
投放正文后、顶部横栏和底部横栏，真实 HTML 中三者顺序正确，`page-bar.js` 只加载一次；
正文 Promotion 的暂停/恢复以及顶部横栏的未开始、有效时间窗、已结束均立即得到正确
结果。站点通过符号链接使用本仓库源码；同一提交生成的 ZIP 另由 Playground、编辑器
E2E 和 Plugin Check 验证。复验结束后临时页面和 Promotion 已删除。

此前五人真实试用已验证单 Promotion 的创建、规则设置、预览和发布流程没有明显摩擦；
它是产品可用性证据，不替代本文件中的自动化、主题和缓存发布门禁。

## 发布步骤

1. 合并仅包含 0.3.1 发布收口的 Pull Request，且所有必需检查通过。
2. 保护 `master`：禁止直接推送和强制推送，要求分支为最新并通过必需检查。
3. 在已验证的 `master` 提交上创建 `v0.3.1` 标签。
4. 等待 Release Gate 完成并创建 GitHub prerelease。
5. 下载 ZIP 与 `.sha256`，在发布页之外重新计算一次 SHA-256 并比对。
6. 仅在真实站点试运行没有出现发布阻断问题后，再决定是否把候选版本标为正式发布。
